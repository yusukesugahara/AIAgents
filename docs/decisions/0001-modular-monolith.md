# ADR-0001: モジュラーモノリスとして開始する

- Status: Accepted
- Date: 2026-07-18

## Context

AIAgentsには、今後用途の異なる複数のAIエージェントを追加する予定です。

エージェントごとにAPI、Worker、DB、認証、監視を分割すると、初期段階からインフラと運用の重複が増えます。一方、すべての処理を1つの巨大なモジュールへ置くと、エージェント間の依存が増え、変更影響を限定できません。

## Decision

初期構成はモジュラーモノリスとします。

- APIは1つ
- Workerは1つ
- PostgreSQLは1つ
- 各エージェントは独立package
- 共通技術機能は`packages`へ配置
- エージェントはRegistryへ登録
- エージェント間の直接参照は禁止

実行単位は共通でも、コード上の境界、DBテーブル、設定、実行履歴はAgent IDで分離します。

## Consequences

### Positive

- 新規エージェント追加時のインフラ作業が少ない
- OAuth、LLM、DB、ログを再利用できる
- ローカル環境をDocker Composeで起動しやすい
- Transactionや実行履歴を一元管理できる
- 必要になったエージェントだけ後から分離できる

### Negative

- 依存ルールを守らないと巨大なモノリスになる
- 1つのWorker障害が複数エージェントへ影響する可能性がある
- エージェントごとのリソース制限は追加実装が必要

## Guardrails

- `apps`、`agents`、`packages`の依存方向をCIで検査する
- エージェントは外部SDKとORMを直接参照しない
- エージェントごとにmanifest、input、output、policy、testsを持つ
- Agent IDごとの実行数、失敗数、処理時間を記録する

## Revisit conditions

次のいずれかが発生した場合は、特定エージェントのサービス分離を検討します。

- 特定エージェントだけ負荷特性が大きく異なる
- 独立したデプロイ頻度が必要
- セキュリティ境界を分ける必要がある
- 障害影響をプロセス単位で分離する必要がある
- 異なるRuntimeや言語が必要
