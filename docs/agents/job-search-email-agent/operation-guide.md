# 就職活動メールエージェント運用ガイド

## 目的

`job-search-email` は、採用・選考・面談調整に関する受信メールを解析し、ユーザーが確認・編集できるGmail下書きを作成します。自動送信はしません。

## 対象となるメール

次のような文脈を持つメールを対象にします。

- 応募、採用、求人、選考、面接、面談、内定
- 候補日時の提示、日程調整、空き時間の確認
- 書類・課題・追加情報の依頼

日程調整メールは、AIが誤って返信不要と返した場合でも返信が必要として扱います。ユーザーが選べるよう、候補日時を入力するプレースホルダーを含む下書きを作成します。

## 実行方法

| 方法 | 用途 | Jobのトリガー |
|---|---|---|
| Worker定期実行 | 新着メールを継続処理 | `schedule` |
| セットアップの「今すぐ定期実行を実行」 | 定期実行と同じ条件で即時確認 | `schedule` |
| 「既存ジョブをリセットして再実行」 | 同一メールを再解析 | `schedule` |
| 最近の受信メールの個別実行 | 指定メールの調査 | `manual` |

定期実行の対象は、次をすべて満たすGoogle接続です。

1. 接続状態が `connected`
2. Gmail読取・Gmail下書き権限を保有
3. 返信下書き設定の作成フラグが有効
4. ユーザー名が設定済み

## 実行結果の読み方

| Step | 意味 |
|---|---|
| `FETCH_EMAIL_THREAD` | Gmailから対象メールとスレッドを取得。件名もここへ保存 |
| `ANALYZE_EMAIL` | 読み取りToolを使ってメール種別、返信要否、面談情報を構造化 |
| `CHECK_REPLY_POLICY` | 下書きToolを許可できるか事前判定 |
| `CHECK_CALENDAR_POLICY` | Calendar登録の安全条件と競合を確認 |
| `CREATE_DRAFT` | Function Callingで安全な下書きToolを実行し、Gmail Draftを作成または再利用 |
| `CREATE_CALENDAR_EVENT` | 必要時のみCalendar予定を作成 |

`CREATE_DRAFT` が成功し、`Gmail Draft ID` が表示された場合、Gmail APIへの作成または既存下書きの確認が完了しています。DBに過去のDraft IDがあっても、Gmailに実在しない場合は再作成します。

Calendar用の追加認可Routeと予定作成処理は実装済みですが、現行セットアップ画面にはCalendar設定フォームがありません。返信設定の保存時は`createCalendarEvents`が`false`になるため、通常の画面操作だけでは`CREATE_CALENDAR_EVENT`は実行されません。

## よくある確認ポイント

### 定期実行の結果が増えない

- 新着メールが検索クエリに一致しているか確認します。
- 同一メールは冪等性により再実行されません。再解析には「既存ジョブをリセットして再実行」を使います。
- Gmail読取・下書き権限と、返信下書き設定を確認します。

### `reply_not_required` で下書きが作られない

返信不要と判断されたメールです。日程調整メールは返信必要に補正されますが、求人文脈のない通常メールは下書きを作成しません。

### `needs_review` になる

安全に返信できないケースです。信頼度不足、返信先ヘッダー不正、必要情報不足、スレッド更新、Calendar競合などを実行履歴の `reviewReason` で確認します。`needs_review`はAgentの業務結果であり、処理自体が正常に完了した場合のJobとRunは`completed`です。

### Gmailに下書きが見つからない

接続したGoogleアカウントと、ブラウザで確認しているGmailアカウントが同じかを確認します。次回の再実行では、DB上のDraft IDだけを信用せず、Gmailに存在するかを確認します。

## データ取り扱い

保存するものは、Job、Run、Step、分類済みの構造化結果、外部リソースID、LLM呼び出しメタデータです。構造化解析結果には判断根拠の`evidence`を最大5件・各240文字まで含めます。メール本文そのもの、Prompt、返信本文、OpenAIへの生データは保存・実行履歴表示しません。
